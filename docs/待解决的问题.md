## Monaco Editor 非基本 ASCII 字符突出显示

**状态**：部分解决

**描述**：Monaco Editor（VS Code 同款编辑器）默认会将中文等非基本 ASCII 字符以黄色高亮标记，在设计上是为了帮助开发者发现可能误输入的不可见字符。但在 Markdown 写作场景下，中文内容占比极高，大面积黄色标记严重影响写作体验。

**已采取措施**：在博客模块的 `MarkdownEditor` 和知识库模块的 `PageEditor` 中，已将 Monaco Editor 的 `unicodeHighlight` 选项设为 `{ nonBasicASCII: false }`，关闭了对非基本 ASCII 字符的高亮。

**残留问题**：部分场景下高亮仍然出现，可能的原因包括：

1. **`ambiguousCharacters` 子选项**：Monaco 还会高亮"歧义字符"（如 Cyrillic 字母 `а` 与拉丁字母 `a` 外形相似），这个未禁用。
2. **`invisibleCharacters` 子选项**：零宽字符等不可见字符的高亮也未禁用。
3. **Overview Ruler 残留**：虽然已设置 `overviewRulerLanes: 0`, `hideCursorInOverviewRuler: true`, `overviewRulerBorder: false`，但不确定是否完全清除。
4. **Minimap 残留**：Minimap 已禁用 (`minimap: { enabled: false }`)，无需处理。

**建议完整配置**：
```ts
unicodeHighlight: {
  nonBasicASCII: false,
  ambiguousCharacters: false,
  invisibleCharacters: false,
}
```

**影响范围**：
- `src/modules/blog/components/MarkdownEditor.tsx`
- `src/modules/knowledge/components/PageEditor.tsx`

**注意**：未来新增"知识库存储源代码"功能后，代码编辑器应恢复 `unicodeHighlight` 的默认行为，仅 Markdown 编辑器需要禁用。

## Dev 版与生产版同时运行导致数据覆盖

**状态**：未解决

**描述**：`requestSingleInstanceLock()` 只能阻止同一可执行文件的重复启动。Dev 版（通过 `electron-vite dev` 启动，跑的是 `node_modules/electron/dist/electron.exe`）和生产版（打包后的独立 `.exe`）被视为不同进程，锁不会生效。

如果同时开着 dev 版和生产版，两个独立的 sql.js 内存副本各自修改数据后分别写盘，后写盘的那个会覆盖先写盘的，导致数据丢失。

**根因**：sql.js 是纯内存数据库，每个进程持有独立的数据库副本，通过 `saveToDisk()` 整库覆盖写回磁盘，不支持增量合并。

**可能方案**：
1. 在 `connection.ts` 中给 `knowledge.db` 加文件锁（如 `flock` / `lockfile`），第二个进程启动时检测锁是否存在，若存在则拒绝启动或弹警告
2. 显式区分 dev 和 production 的 `userData` 路径，让两者使用不同的数据库文件
3. 改用 better-sqlite3 等原生 SQLite 库，利用 WAL 模式实现多进程安全访问（代价是放弃 sql.js 的跨平台便利性）

**影响范围**：
- `electron/database/connection.ts`
- `electron/main/index.ts`
